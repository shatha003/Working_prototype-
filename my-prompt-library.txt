3. Rules Engine Developer (Triage Logic)
──────────────────────────────────────────
Prompt:

"You are an expert engineer in a fast-paced coding competition. Speed,
readability, and functional correctness are my top priorities. Avoid
unnecessary abstractions unless they are critical for the functionality.

Based on the approved architecture, implement the triage engine logic in
src/engine/triage.js and the supporting data in src/data/requests.js and
src/engine/knowledgeBase.js. This is the core of the prototype — it must be
fast, correct, and transparent.

The engine receives 5 requests and returns a decision object for each:

const decision = {
  id, customer, type, priority, priorityScore, confidence,
  recommendedAction, owner, autoResolvable, needsManager,
  suggestedResponse, reason, evidence
}

Scoring model:
- urgency (0-30): bad-review threats, VIP churn, angry customers score high
- customerValue (0-20): VIP boost, new-customer modest
- resolutionRisk (0-20): low risk = higher score; refund/ambiguous = lower
- slaBreachRisk (0-15): reputation threat, time-sensitive
- policyFit (0-15): KB match, slots available, refund within limit

Confidence is computed per recommended action, not globally. It reflects
"how sure the AI is the action is correct." Range: 0.00 to 0.96.

Implement step-by-step:
1. Define the 5 requests with realistic detail in requests.js
2. Define KB entries in knowledgeBase.js (pricing, booking policy, VIP policy)
3. Implement the scoring function in triage.js
4. Implement the decision logic (assign action, owner, response, reason)
5. Enforce the refund-limit edge case (confidence = 0 for auto-resolve)
6. Export triageAll() and triageOne() functions

Each function must be under 30 lines. No classes — use pure functions.
Refer to @tech-specs.md for coding standards."

7. Workflow Automation Developer (Agentic Pipeline)
─────────────────────────────────────────────────────
Prompt:

"You are an expert engineer in a fast-paced coding competition. Speed,
readability, and functional correctness are my top priorities. Avoid
unnecessary abstractions unless they are critical for the functionality.

Based on the approved architecture, implement the automated workflow system
that runs after the triage engine executes. The workflow must:

1. Read the decisions array from triageAll()
2. For each request where autoResolvable === true AND confidence > 0.85:
   - Assign the request to an available staff member (decrement staff count)
   - If the request involves a booking, decrement slots available
   - Generate the suggested response from KB or template
   - Mark status as "Auto-resolved"
   - Push a step to the trace log array
3. For requests where needsManager === true:
   - Mark status as "Escalated"
   - Push escalation step to trace log
   - Ensure Send button is disabled in the UI (handled via state)
4. For remaining requests:
   - Mark status as "Pending human review"
   - Assign to the remaining staff member if any
   - Push pending step to trace log

The trace log is an array of { timestamp, requestId, action, detail } objects
displayed in AutomationsPanel as a scrollable timeline. Each step should read
like: "Detected: pricing query → KB match → Confidence 0.96 → Drafted response
→ Status: Auto-resolved"

Refer to @tech-specs.md for coding standards. Keep all workflow logic in
a single src/engine/workflow.js file that exports runWorkflow(decisions, state)."
